# trace-pack v0.2 — Multi-app generalization (design)

> **Status: shipped (2026-05-23/24).** trace-pack now implements this design — generalized event core +
> cost (#19), rollup status/spend + multi-source queries (#20), and the multi-source dashboard UI (#21);
> prod migrated (`004`/`005`). Both consumers are wired (ask-zeroindex + contract-lens). Companion to
> `PROJECT.md` §11. Decisions confirmed 2026-05-22 (§8); consumer token-usage readiness verified (§6.1).
> **Deferred:** the admin events-table + `/admin/[id]` per-event-type generalization (§5.4), pending the
> first real non-`ask` events.

---

## 1. The core problem

v0.1 conflates two things in one schema:

1. **The universal envelope** — "an LLM request happened in app X, at time T, on model M, costing C, and it
   succeeded or failed." Every Claude app has this.
2. **The RAG shape of `ask-zeroindex`** — `question`, `retrievedIds`, `citationCount`, `retrievalMs`,
   `firstTokenMs`, and the outcome vocabulary `retrieval_failed`/`stream_failed`. Only a streaming RAG Q&A app has this.

The ingest schema hard-codes #2: `event: z.literal('ask')` (`src/ingest/schema.ts:9`), required RAG fields,
a RAG outcome enum, and a `question_hash`-based idempotency key. The dashboards (citation histogram, top
retrieved chunks, first-token latency) are RAG-specific too. The other ZeroIndex apps aren't RAG Q&A:

| App               | What it does                             | Why the `ask` shape doesn't fit                                                                  |
| ----------------- | ---------------------------------------- | ------------------------------------------------------------------------------------------------ |
| **contract-lens** | PDF field extraction w/ cited confidence | No question/retrieval/first-token; outcomes are extraction success / low-confidence / parse-fail |
| **intake-zero**   | document→extract→email pipeline          | Multi-step workflow, not one Q&A turn                                                            |
| **repo-xray**     | agentic codebase analysis (two-phase)    | Findings + phases, not retrieval + citations                                                     |

**What's already fine:** storage is multi-_source_ today — `events.source`, source-scoped indexes,
per-source bearer auth (`SOURCE_TOKEN_<NAME>`), and a `DISTINCT source` query (`rollup.ts:39`). The data can
already hold many senders. What's missing is a schema that models _any_ app shape, a universal cost axis, and
a UI that shows more than one source.

**v0.2 thesis:** separate the universal **core** from the per-event-type **extension**, add the one axis every
Claude app shares (**cost / tokens**), and make the UI source-aware.

## 2. Generalized event model

### 2.1 Common core (typed columns, every event)

| Field            | v0.1 status         | v0.2                                                                                                                                                     |
| ---------------- | ------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `source`         | ✅ exists           | unchanged (tenant key)                                                                                                                                   |
| `event`          | `literal('ask')`    | open discriminator — `ask`, `extract`, `pipeline`, `analyze`, `generic`, …                                                                               |
| `ts`             | ✅ exists           | unchanged                                                                                                                                                |
| `model`          | ✅ exists           | unchanged (nullable — some events aren't a single model call)                                                                                            |
| `status`         | ❌ (only `outcome`) | **NEW** — universal `ok` / `error` / `aborted`. The cross-app success axis.                                                                              |
| `outcome_reason` | ❌                  | **NEW** nullable — app-specific reason string (`retrieval_failed`, `stream_failed`, `extraction_low_confidence`, …). Replaces overloading the flat enum. |
| `total_ms`       | ✅ exists           | unchanged (wall-clock; universal)                                                                                                                        |
| `input_tokens`   | ❌                  | **NEW** nullable int                                                                                                                                     |
| `output_tokens`  | ❌                  | **NEW** nullable int                                                                                                                                     |
| `cost_usd`       | ❌                  | **NEW** nullable real — derived at ingest from a model→price table (see §4)                                                                              |
| `error_message`  | ✅ exists           | unchanged                                                                                                                                                |
| `raw_json`       | ✅ exists           | unchanged (the forward-compat escape hatch)                                                                                                              |

### 2.2 Why `status` + `outcome_reason` instead of growing the enum

Adding `extraction_low_confidence`, `parse_failed`, etc. to the flat `outcome` enum would make every cross-app
chart special-case which values count as "failure." A two-field split keeps a clean universal axis (`status`)
for the multi-app overview + outcome charts, while preserving the rich per-app reason for the detail/error views.

**Back-compat mapping** (applied in the migration backfill and at ingest for `ask`):

| v0.1 `outcome`     | → `status` | → `outcome_reason` |
| ------------------ | ---------- | ------------------ |
| `ok`               | `ok`       | `null`             |
| `aborted`          | `aborted`  | `null`             |
| `retrieval_failed` | `error`    | `retrieval_failed` |
| `stream_failed`    | `error`    | `stream_failed`    |

### 2.3 Extension fields — hybrid, not speculative

The `ask` RAG columns (`question`, `question_hash`, `retrieved_ids`, `citation_count`, `retrieval_ms`,
`first_token_ms`) **stay as typed columns** — they already exist, are indexed, and ask-zeroindex depends on them.
They are re-framed as the **`ask` extension**, not core.

For _new_ event types, fields land in `raw_json` first (the detail page already shows the full `raw_json`
block, so unmapped fields stay visible). A field is promoted to a typed column **only when a chart needs to
aggregate it** — not before.

> **Decision (anti-speculation):** do NOT design `extract`/`pipeline`/`analyze` extension columns now. Define
> them when that app is actually instrumented and we know its real fields — design for fields a consumer
> actually emits, not speculative ones. v0.2 ships the core + cost + UI; the second app's extension is
> designed at onboarding time.

### 2.4 Idempotency key

v0.1 dedup is `UNIQUE (source, ts, question_hash)`, and `question_hash` is `NOT NULL`. Non-RAG events have no
question.

The v0.1 naming is the leaky abstraction: `question_hash` was never conceptually "the hash of the question" —
it's **the thing we dedup on**, which for `ask` is _derived from_ the question. So the fix is not to keep that
name and stuff foreign data into it (that re-overloads a column whose name no longer matches its data); the fix
is to **name the column for its actual role**.

- **(A) Rename `question_hash → dedup_hash` — recommended.** Pure `ALTER TABLE ... RENAME COLUMN` (SQLite
  ≥3.25, supported by libsql); the dependent UNIQUE constraint and `idx_events_source_hash` index references
  update automatically — **no table rebuild, no constraint surgery.** `dedup_hash` stays `NOT NULL` for every
  type: `ask` sets it to `sha256(question)` (existing rows + values untouched), other types set it to
  `sha256(idempotencyKey ?? canonical(raw_json))`. The name now describes the role across all types. The
  "question clusters" admin view still works for `ask` (identical questions hash identically) and stays gated
  to `ask`. One column, honest name, zero redundancy.
- **(B) Full split — `dedup_hash` core + nullable `question_hash` ask-extension.** The purest
  inheritance/extension form: `dedup_hash` is core (every event), `question_hash` becomes an explicit nullable
  `ask`-only column. But it needs the SQLite 12-step **table rebuild** (can't drop `NOT NULL` or move a UNIQUE
  constraint via `ALTER`: create `events_new`, copy with column mapping, drop, rename), and it stores the same
  hash twice for every `ask` row (`dedup_hash == question_hash`). More ceremony + mild redundancy for a
  marginal explicitness gain.

**Recommendation: (A).** The rename de-overloads the column (the honest-naming win) without the rebuild — and
(B)'s duplicate-hash-per-ask-row makes it slightly _worse_ here, not better. Revisit (B) only if `ask` ever
needs a question hash that legitimately differs from its dedup key.

## 3. Cost / token tracking (the highest-value addition)

Every Claude app — RAG or not — has a spend. trace-pack captures none today. v0.2 adds it as core.

- Consumers emit optional `inputTokens` + `outputTokens` and — when prompt caching is in play —
  `cacheCreationInputTokens` + `cacheReadInputTokens` (additive; apps that don't send them get `null` cost,
  not a rejection).
- trace-pack **owns** the pricing table (`src/lib/pricing.ts`): `model → { inputPer1M, outputPer1M }` plus the
  Anthropic cache multipliers (cache **write** ≈ 1.25× base input, cache **read** ≈ 0.1×). Pricing four token
  classes — not just input/output — matters because both inspected consumers already run prompt caching
  (see §6.1), so flat input+output pricing would misstate real cost.
- Cost is computed **at ingest** and stored in `cost_usd`, so rollups sum a stored number and a pricing change
  doesn't require re-reading raw events. (Trade-off: stored cost is non-retroactive — a price-table edit
  doesn't restate history. Acceptable; noted. The alternative — compute at read — keeps pricing always-current
  but makes every dashboard query do per-row arithmetic over a window. Not worth it at this scale.)
- Unknown model → `cost_usd = null`, tokens still stored (so the gap is visible, not silently zero).

## 4. Rollup changes

`rollup_daily` gains: `n_ok`, `n_error`, `n_aborted` (status counts), `sum_cost_usd`, `sum_input_tokens`,
`sum_output_tokens`. The existing RAG columns (`ok`/`retrieval_failed`/`stream_failed`/`aborted`,
`p*_first_token_ms`, `avg_citations`) stay for `ask` and are simply `0`/`null` for sources that don't emit them.
`aggregateSource` (`rollup.ts:45`) counts by `status` generically and only fills the RAG metrics when those
columns are present in the window. History is restated by replaying `GET /api/rollup?day=<d>` across the window
(the endpoint already supports `?day=`).

## 5. UI surfaces

### 5.1 Source switching

- `?source=<name>` on `/` and `/admin`. `/admin` already reads async `searchParams` (for `page`/`outcome`);
  `/` gains a `searchParams` arg. Absent → `DEFAULT_SOURCE`.
- Promote the `DISTINCT source` query to a shared `listSources()` helper; render a switcher (a pill strip like
  the existing outcome filter).
- `?source=` first; pretty `/s/<source>` URLs are a later nicety, not a v0.2 blocker.

### 5.2 Multi-app overview (the "what are all my Claude apps doing" view)

A new mode (`?source=all`, or the default landing when >1 source exists): **one row per source** —
events (24h / 30d), error rate (`status=error` / total), p95 `total_ms`, total `cost_usd` (30d), last-seen ts;
each row links into that source's dashboard. This is the headline "all my Claude apps at a glance" view.

### 5.3 Chart gating (core vs RAG)

| Chart                            | Scope               |
| -------------------------------- | ------------------- |
| Traffic                          | core — every source |
| Outcomes (by `status`)           | core                |
| Latency (`total_ms` p50/p95/p99) | core                |
| **Spend (cost + tokens)**        | core — **new**      |
| First-token latency              | RAG-only            |
| Citation histogram               | RAG-only            |
| Top retrieved chunks             | RAG-only            |

Gating is **derived from data presence** (extend the existing `hasData` pattern: "does this source have
non-null `first_token_ms` / `citation_count` in the window?") — no static per-app registry to maintain.

### 5.4 Admin

- **Shipped:** source switcher on `/admin` (switches the per-source events / errors / clusters).
- **Deferred** — the events-table and detail-page generalizations below are intentionally _not_ built yet.
  Shaping per-event-type columns is speculative until a non-`ask` consumer's events are actually landing, so
  this rides with the first real non-`ask` source (contract-lens):
  - **Events table.** Still ask-shaped (Total / First token / Cites / Question) — a non-`ask` row renders `—`
    in the RAG columns. Intended end state: core columns always (id, ts, status, total_ms, cost), RAG columns
    only for `ask`, and the trailing column a generic summary (the event-type's primary `raw_json` field, or
    `outcome_reason`) for other types.
  - **`/admin/[id]` detail page.** Its `raw_json` block already renders any payload, but the typed kv-list is
    still ask-shaped and omits the v0.2 core fields (`status`, `cost_usd`, `input`/`output_tokens`,
    `outcome_reason`). Generalizing the kv-list rides with the events-table reshape above.

## 6. Ingest contract changes (consumer impact)

**ask-zeroindex requires zero change.** It keeps POSTing its current `{ event: "ask", outcome, … }` body;
trace-pack maps `outcome → status + outcome_reason` server-side and accepts the RAG fields as the `ask`
extension.

For trace-pack:

- `event` becomes an open enum / discriminated union instead of `literal('ask')`.
- New **optional** core fields accepted: `inputTokens`, `outputTokens`, `cacheCreationInputTokens`,
  `cacheReadInputTokens`, and (for non-`ask` types) `status` + `outcomeReason` and an optional
  `idempotencyKey`.
- Per-source bearer auth is unchanged (already multi-source).
- `passthrough()` stays — unknown fields still land in `raw_json`.

A new event type onboards by: (1) picking its `event` name + `status` mapping, (2) sending the core fields +
whatever extension fields it has (into `raw_json`), (3) optionally promoting an extension field to a typed
column once a chart needs it.

### 6.1 Consumer readiness — token usage (inspected 2026-05-22)

The cost core assumes each consumer has Anthropic `usage` in hand where it emits. Verified:

- **contract-lens — ready, emit-wiring only.** `src/lib/extract.ts` already captures the full breakdown
  (`input_tokens`, `output_tokens`, `cache_creation_input_tokens`, `cache_read_input_tokens`) from the
  non-streaming `messages.create` response into its extraction metadata (and even renders it in the UI). It
  has **no** trace-pack emitter yet, so onboarding it is purely "build the fire-and-forget POST" — no usage
  instrumentation needed. This is why it's the right first onboard.
- **ask-zeroindex — available, small capture change.** It streams via the SDK `MessageStream`, and the route
  already iterates the stream events, so `usage` is reachable (from `message_start` / `message_delta`, or
  `finalMessage()` after the loop). But `logAsk`'s `AskTrace` doesn't carry tokens today, so wiring cost means
  a small change in that repo: capture usage in the route + add the token fields to `AskTrace`. Failed/aborted
  streams may have partial/no usage → `cost_usd = null` for those, which is fine.

Both already run **prompt caching**, which is why §3 prices cache write/read separately rather than flat
input/output.

## 9. What this explicitly does NOT do

- No per-app speculative schemas (extension fields are designed at onboarding, §2.3).
- No table rebuild — the `question_hash → dedup_hash` change is a pure column rename (§2.4); the full
  rebuild-based split is explicitly deferred.
- No alerting, no live-tail, no embedding-based clustering — still `PROJECT.md` §11 future work.
- No auth change — `/admin` stays single-owner basic-auth (multi-_tenant_ with separate logins is a different,
  later project; this is multi-_source_ under one owner).
