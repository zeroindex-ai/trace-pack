# trace-pack v0.2 ‚Äî Multi-app generalization (design)

> **Status: design / not yet implemented.** This doc proposes how `trace-pack` graduates from
> single-app (`ask-zeroindex`-shaped) to true multi-app observability for any Claude-based
> ZeroIndex service. Companion to `PROJECT.md` ¬ß11 (v0.2 candidate work). No code has been written
> against this yet ‚Äî it exists to lock the schema + UI decisions **before** a migration touches prod.

---

## 1. The core problem

v0.1 conflates two things in one schema:

1. **The universal envelope** ‚Äî "an LLM request happened in app X, at time T, on model M, costing C, and it
   succeeded or failed." Every Claude app has this.
2. **The RAG shape of `ask-zeroindex`** ‚Äî `question`, `retrievedIds`, `citationCount`, `retrievalMs`,
   `firstTokenMs`, and the outcome vocabulary `retrieval_failed`/`stream_failed`. Only a streaming RAG Q&A app has this.

The ingest schema hard-codes #2: `event: z.literal('ask')` (`src/ingest/schema.ts:9`), required RAG fields,
a RAG outcome enum, and a `question_hash`-based idempotency key. The dashboards (citation histogram, top
retrieved chunks, first-token latency) are RAG-specific too. The other ZeroIndex apps aren't RAG Q&A:

| App | What it does | Why the `ask` shape doesn't fit |
| --- | --- | --- |
| **contract-lens** | PDF field extraction w/ cited confidence | No question/retrieval/first-token; outcomes are extraction success / low-confidence / parse-fail |
| **intake-zero** | document‚Üíextract‚Üíemail pipeline | Multi-step workflow, not one Q&A turn |
| *agentic-analysis shape* | multi-step, tool-driven analysis | Findings + phases, not retrieval + citations |

**What's already fine:** storage is multi-*source* today ‚Äî `events.source`, source-scoped indexes,
per-source bearer auth (`SOURCE_TOKEN_<NAME>`), and a `DISTINCT source` query (`rollup.ts:39`). The data can
already hold many senders. What's missing is a schema that models *any* app shape, a universal cost axis, and
a UI that shows more than one source.

**v0.2 thesis:** separate the universal **core** from the per-event-type **extension**, add the one axis every
Claude app shares (**cost / tokens**), and make the UI source-aware.

## 2. Generalized event model

### 2.1 Common core (typed columns, every event)

| Field | v0.1 status | v0.2 |
| --- | --- | --- |
| `source` | ‚úÖ exists | unchanged (tenant key) |
| `event` | `literal('ask')` | open discriminator ‚Äî `ask`, `extract`, `pipeline`, `analyze`, `generic`, ‚Ä¶ |
| `ts` | ‚úÖ exists | unchanged |
| `model` | ‚úÖ exists | unchanged (nullable ‚Äî some events aren't a single model call) |
| `status` | ‚ùå (only `outcome`) | **NEW** ‚Äî universal `ok` / `error` / `aborted`. The cross-app success axis. |
| `outcome_reason` | ‚ùå | **NEW** nullable ‚Äî app-specific reason string (`retrieval_failed`, `stream_failed`, `extraction_low_confidence`, ‚Ä¶). Replaces overloading the flat enum. |
| `total_ms` | ‚úÖ exists | unchanged (wall-clock; universal) |
| `input_tokens` | ‚ùå | **NEW** nullable int |
| `output_tokens` | ‚ùå | **NEW** nullable int |
| `cost_usd` | ‚ùå | **NEW** nullable real ‚Äî derived at ingest from a model‚Üíprice table (see ¬ß4) |
| `error_message` | ‚úÖ exists | unchanged |
| `raw_json` | ‚úÖ exists | unchanged (the forward-compat escape hatch) |

### 2.2 Why `status` + `outcome_reason` instead of growing the enum

Adding `extraction_low_confidence`, `parse_failed`, etc. to the flat `outcome` enum would make every cross-app
chart special-case which values count as "failure." A two-field split keeps a clean universal axis (`status`)
for the multi-app overview + outcome charts, while preserving the rich per-app reason for the detail/error views.

**Back-compat mapping** (applied in the migration backfill and at ingest for `ask`):

| v0.1 `outcome` | ‚Üí `status` | ‚Üí `outcome_reason` |
| --- | --- | --- |
| `ok` | `ok` | `null` |
| `aborted` | `aborted` | `null` |
| `retrieval_failed` | `error` | `retrieval_failed` |
| `stream_failed` | `error` | `stream_failed` |

### 2.3 Extension fields ‚Äî hybrid, not speculative

The `ask` RAG columns (`question`, `question_hash`, `retrieved_ids`, `citation_count`, `retrieval_ms`,
`first_token_ms`) **stay as typed columns** ‚Äî they already exist, are indexed, and ask-zeroindex depends on them.
They are re-framed as the **`ask` extension**, not core.

For *new* event types, fields land in `raw_json` first (the detail page already renders any payload as a
kv-list). A field is promoted to a typed column **only when a chart needs to aggregate it** ‚Äî not before.

> **Decision (anti-speculation):** do NOT design `extract`/`pipeline`/`analyze` extension columns now. Define
> them when that app is actually instrumented and we know its real fields ‚Äî design for fields a consumer
> actually emits, not speculative ones. v0.2 ships the core + cost + UI; the second app's extension is
> designed at onboarding time.

### 2.4 Idempotency key

v0.1 dedup is `UNIQUE (source, ts, question_hash)`, and `question_hash` is `NOT NULL`. Non-RAG events have no
question.

The v0.1 naming is the leaky abstraction: `question_hash` was never conceptually "the hash of the question" ‚Äî
it's **the thing we dedup on**, which for `ask` is *derived from* the question. So the fix is not to keep that
name and stuff foreign data into it (that re-overloads a column whose name no longer matches its data); the fix
is to **name the column for its actual role**.

- **(A) Rename `question_hash ‚Üí dedup_hash` ‚Äî recommended.** Pure `ALTER TABLE ... RENAME COLUMN` (SQLite
  ‚â•3.25, supported by libsql); the dependent UNIQUE constraint and `idx_events_source_hash` index references
  update automatically ‚Äî **no table rebuild, no constraint surgery.** `dedup_hash` stays `NOT NULL` for every
  type: `ask` sets it to `sha256(question)` (existing rows + values untouched), other types set it to
  `sha256(idempotencyKey ?? canonical(raw_json))`. The name now describes the role across all types. The
  "question clusters" admin view still works for `ask` (identical questions hash identically) and stays gated
  to `ask`. One column, honest name, zero redundancy.
- **(B) Full split ‚Äî `dedup_hash` core + nullable `question_hash` ask-extension.** The purest
  inheritance/extension form: `dedup_hash` is core (every event), `question_hash` becomes an explicit nullable
  `ask`-only column. But it needs the SQLite 12-step **table rebuild** (can't drop `NOT NULL` or move a UNIQUE
  constraint via `ALTER`: create `events_new`, copy with column mapping, drop, rename), and it stores the same
  hash twice for every `ask` row (`dedup_hash == question_hash`). More ceremony + mild redundancy for a
  marginal explicitness gain.

**Recommendation: (A).** The rename de-overloads the column (the honest-naming win) without the rebuild ‚Äî and
(B)'s duplicate-hash-per-ask-row makes it slightly *worse* here, not better. Revisit (B) only if `ask` ever
needs a question hash that legitimately differs from its dedup key.

## 3. Cost / token tracking (the highest-value addition)

Every Claude app ‚Äî RAG or not ‚Äî has a spend. trace-pack captures none today. v0.2 adds it as core.

- Consumers emit optional `inputTokens` + `outputTokens` (additive; apps that don't send them get `null` cost,
  not a rejection).
- trace-pack **owns** the pricing table (`src/lib/pricing.ts`): `model ‚Üí { inputPer1M, outputPer1M }`. Cost is
  computed **at ingest** and stored in `cost_usd`, so rollups sum a stored number and a pricing change doesn't
  require re-reading raw events. (Trade-off: stored cost is non-retroactive ‚Äî a price-table edit doesn't
  restate history. Acceptable; noted. The alternative ‚Äî compute at read ‚Äî keeps pricing always-current but
  makes every dashboard query do per-row arithmetic over a window. Not worth it at this scale.)
- Unknown model ‚Üí `cost_usd = null`, tokens still stored (so the gap is visible, not silently zero).

## 4. Rollup changes

`rollup_daily` gains: `n_ok`, `n_error`, `n_aborted` (status counts), `sum_cost_usd`, `sum_input_tokens`,
`sum_output_tokens`. The existing RAG columns (`ok`/`retrieval_failed`/`stream_failed`/`aborted`,
`p*_first_token_ms`, `avg_citations`) stay for `ask` and are simply `0`/`null` for sources that don't emit them.
`aggregateSource` (`rollup.ts:45`) counts by `status` generically and only fills the RAG metrics when those
columns are present in the window. History is restated by replaying `GET /api/rollup?day=<d>` across the window
(the endpoint already supports `?day=`).

## 5. UI surfaces

### 5.1 Source switching
- `?source=<name>` on `/` and `/admin`. `/admin` already reads async `searchParams` (for `page`/`outcome`);
  `/` gains a `searchParams` arg. Absent ‚Üí `DEFAULT_SOURCE`.
- Promote the `DISTINCT source` query to a shared `listSources()` helper; render a switcher (a pill strip like
  the existing outcome filter).
- `?source=` first; pretty `/s/<source>` URLs are a later nicety, not a v0.2 blocker.

### 5.2 Multi-app overview (the "what are all my Claude apps doing" view)
A new mode (`?source=all`, or the default landing when >1 source exists): **one row per source** ‚Äî
events (24h / 30d), error rate (`status=error` / total), p95 `total_ms`, total `cost_usd` (30d), last-seen ts;
each row links into that source's dashboard. This is the headline "all my Claude apps at a glance" view.

### 5.3 Chart gating (core vs RAG)
| Chart | Scope |
| --- | --- |
| Traffic | core ‚Äî every source |
| Outcomes (by `status`) | core |
| Latency (`total_ms` p50/p95/p99) | core |
| **Spend (cost + tokens)** | core ‚Äî **new** |
| First-token latency | RAG-only |
| Citation histogram | RAG-only |
| Top retrieved chunks | RAG-only |

Gating is **derived from data presence** (extend the existing `hasData` pattern: "does this source have
non-null `first_token_ms` / `citation_count` in the window?") ‚Äî no static per-app registry to maintain.

### 5.4 Admin
- Source switcher.
- Events table: core columns always (id, ts, status, total_ms, cost); the RAG columns (cites, first-token,
  question) render for `ask`. For other event types, the trailing "Question" column becomes a generic
  "summary" column (the event-type's primary field from `raw_json`, or the `outcome_reason`).
- The `/admin/[id]` detail page is already generic (typed kv-list + full `raw_json`) ‚Äî minimal change.

## 6. Ingest contract changes (consumer impact)

**ask-zeroindex requires zero change.** It keeps POSTing its current `{ event: "ask", outcome, ‚Ä¶ }` body;
trace-pack maps `outcome ‚Üí status + outcome_reason` server-side and accepts the RAG fields as the `ask`
extension.

For trace-pack:
- `event` becomes an open enum / discriminated union instead of `literal('ask')`.
- New **optional** core fields accepted: `inputTokens`, `outputTokens`, and (for non-`ask` types) `status` +
  `outcomeReason` and an optional `idempotencyKey`.
- Per-source bearer auth is unchanged (already multi-source).
- `passthrough()` stays ‚Äî unknown fields still land in `raw_json`.

A new event type onboards by: (1) picking its `event` name + `status` mapping, (2) sending the core fields +
whatever extension fields it has (into `raw_json`), (3) optionally promoting an extension field to a typed
column once a chart needs it.

## 9. What this explicitly does NOT do

- No per-app speculative schemas (extension fields are designed at onboarding, ¬ß2.3).
- No table rebuild ‚Äî the `question_hash ‚Üí dedup_hash` change is a pure column rename (¬ß2.4); the full
  rebuild-based split is explicitly deferred.
- No alerting, no live-tail, no embedding-based clustering ‚Äî still `PROJECT.md` ¬ß11 future work.
- No auth change ‚Äî `/admin` stays single-owner basic-auth (multi-*tenant* with separate logins is a different,
  later project; this is multi-*source* under one owner).
